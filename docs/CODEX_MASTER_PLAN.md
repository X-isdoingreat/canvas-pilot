# Codex Master Plan

This document defines the Codex sidecar as a file-driven execution system.
It is the top-level plan Codex should follow when doing long-running Canvas
Pilot work in this repository.

## One-Line Architecture

Codex runs one small batch at a time from `docs/CODEX_BATCHES.md`, executes
only the files allowed by that batch, proves the result with acceptance and
regression checks, writes a report, then marks the batch complete.

## Design Goal

The goal is not to make Codex "remember to be careful". The goal is to make
the repo itself constrain Codex:

```text
AGENTS.md
  -> permanent repo rules and driver boundaries

docs/CODEX_MASTER_PLAN.md
  -> long-term architecture and execution model

docs/CODEX_CC_SYNC_WORKFLOW.md
  -> CC-to-Codex resync process after Claude driver changes

docs/CODEX_BATCHES.md
  -> ordered work queue, one active batch at a time

docs/CODEX_TASKS.md
  -> task ledger for the current implementation surface

docs/CODEX_ACCEPTANCE.md
docs/CODEX_REGRESSION.md
scripts/codex_check.py
  -> machine-checkable done definition

.codex/hooks/*
  -> continuation guard when Codex tries to stop too early

runs/codex/<date>/*.md
  -> evidence trail for what was done and verified
```

## Builder Pattern

This follows the current mainstream builder pattern for long agent tasks:

1. Put durable repo rules in `AGENTS.md`.
2. Put repeatable workflows in skills.
3. Put long-horizon execution in a plan or batch ledger.
4. Put "done" behind scripts, tests, and reports.
5. Use hooks to block early stop or unsafe actions.
6. Use CI or `codex exec` later only after the local loop is stable.

## Control Plane

`docs/CODEX_BATCHES.md` is the control plane.

Each batch must define:

- `status`
- `parity_ids`
- `allowed_files`
- `acceptance`
- `regression`
- `done_when`
- `report`

Only one batch may be `in_progress`.

Status meanings:

| Status | Meaning |
|---|---|
| `next` | ready for Codex to claim |
| `in_progress` | current active batch |
| `done` | verified and reported |
| `blocked` | failed check or missing decision |
| `later` | intentionally deferred |

## Execution Loop

Codex must use this loop for every implementation batch:

1. Read `AGENTS.md`.
2. Read this file.
3. Read `docs/CODEX_BATCHES.md`.
4. If no batch is `in_progress`, claim the first `next` batch by changing it
   to `in_progress`.
5. Read that batch's allowed files and checks.
6. Make only the allowed edits.
7. Run acceptance checks.
8. Run regression checks.
9. Write `runs/codex/<YYYY-MM-DD>/PARITY_<batch>.md`.
10. Update `docs/CODEX_TASKS.md` and `docs/CODEX_PARITY_MATRIX.md`.
11. Mark the batch `done` only after all checks pass.

If any check fails, leave the batch `blocked`, write the report, and stop.

## Stop Hook Rule

The Stop hook is the enforcement layer. It should block stopping when:

- a batch is `in_progress` and no matching report exists
- a batch is marked `done` but its parity rows are not done
- execute-mode work has missing or invalid `result.json`
- a required acceptance or regression marker is missing from the report

The hook should tell Codex exactly which file or check is missing.

## Skill Strategy

Codex skills mirror the Claude driver shape, but they are not copied into the
Claude driver and do not modify `.claude/`.

Planned Codex skills:

| Skill | Purpose |
|---|---|
| `canvas-bootstrap` | first-run course fingerprinting and public-safe skill skeleton generation |
| `canvas-scan` | auth probe, pending scan, plan generation, stop |
| `canvas-execute` | approval parse, dispatch, result ledger, report |
| `canvas-skip` | conservative fallback for unsupported/manual/deferred items |
| `canvas-review` | future fresh-eye verification before submit or delivery |

Course-specific skills generated by Codex must live under `.agents/skills/`.
They must not copy private Claude course skills or private IDs.

## Runtime Sequence For A Student

The target public student flow is:

```text
setup auth
  -> API token path or cookie path

canvas-bootstrap
  -> scan active courses
  -> show recurring patterns
  -> student names per-course skills
  -> write .agents/skills/canvas-<name>/SKILL.md skeletons
  -> update routes config

student fills skeleton TODOs
  -> remove UNFILLED_SKELETON sentinel

canvas-scan
  -> write assignments.json and plan.json
  -> render due table
  -> stop

student approves

canvas-execute
  -> dispatch approved items
  -> write result.json per item
  -> update _processed.json
  -> write REPORT.md
```

Default behavior is draft-only. Submission requires explicit authorization and
passing verification.

## Batch Roadmap

The Codex sidecar should move in this order:

| Batch | Goal |
|---|---|
| B1 | docs and repo skill skeletons |
| B2 | basic hooks and hook tests |
| B3 | `canvas-bootstrap` public-safe first-run flow |
| B4 | real `canvas-scan` behavior parity |
| B5 | real `canvas-execute` behavior parity |
| B6 | runtime fixtures and fake Canvas regression suite |
| B7 | advanced hook parity: spec grounding, identifier grounding, pre-submit gate |
| B8 | first-student onboarding for API token and cookie auth |
| B9 | optional plugin packaging |
| B10 | optional CI / `codex exec` automation |

## CC Update Sync

When the Claude Code driver changes, run:

```powershell
python scripts\codex_cc_sync_plan.py
```

This writes `runs/codex/<date>/CC_SYNC_PLAN.md`. Codex should use that report
to update the public-safe Claude baseline, parity matrix, and batch queue.
After the relevant Codex batch passes, accept the current Claude driver
snapshot with:

```powershell
python scripts\codex_cc_sync_plan.py --update-baseline
```

Interactive Codex sessions should use `$cc-sync-runner`, which wraps this
process in the batch/check/report loop.

If the user asks to continue automatically, `$cc-sync-runner` may continue past
the CC diff into the Codex roadmap. It must still work one batch at a time and
stop on permissions, private data, live Canvas actions, out-of-scope edits, or
repeated unresolved check failures.

## Done Definition

Codex sidecar is "usable for student pilot" only when:

- bootstrap can create public-safe per-course skill skeletons
- scan can produce a real approval plan from configured auth
- execute can process approved items without touching unapproved items
- every item gets a valid `result.json`
- `REPORT.md` clearly states draft-ready, skipped, error, and urgent items
- hooks prevent early stop during execute
- the full check command for the active batch passes
- private Claude driver files remain untouched

## Non-Goals

- Do not merge Claude and Codex drivers.
- Do not convert private Claude course skills automatically.
- Do not submit student work by default.
- Do not treat Canvas assignment descriptions as complete specs.
- Do not rely on prose-only instructions where a check or hook can enforce the rule.
