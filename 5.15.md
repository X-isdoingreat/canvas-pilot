# 5.15 — Canvas Pilot v0.1 到底要做出什么样的产品

这份是从 fork 用户视角看 v0.1 ship 之后跟今天的 `public-main` 比，产品能干什么新事、不再干什么旧事、以及关键产品决定。

---

## 今天的 public-main 长什么样

任何人现在 clone `X-isdoingreat/Canvas_pilot` 下来跑：

- 看 Canvas 上的待办、做一个审批列表、按学生选的项目分发——五个框架级 skill 实际可用
- 但分发到"代码课 / 写作课 / zyBook 课 / quiz 课"任一类时，对应的 skill 是空壳——它会停下来说"请给我你课的配置文件，我才能动"
- 学生没地方拿到这个配置文件的模板，也没人告诉他该怎么写
- 产品只完成了"分诊"，治疗手段都是空的

fork 用户体感：**像 demo 不像真产品**。

---

## v0.1 ship 之后产品多了什么

### Bootstrap 从问卷式变成侦探式

这是 v0.1 最大的设计转向。

旧 mental model：bootstrap 是一份采访问卷——"你课叫什么？老师 spec 贴哪？作业怎么命名？……" 学生答完一长串问题，系统才能动。

新 mental model：**bootstrap 是个会读 Canvas 的侦探**。学生选了要追踪的课之后，系统主动 probe：
- 课名直接从 Canvas API 拿
- 课类型（代码 / 写作 / zyBook / quiz）通过看几个最近作业的 description / attachments / name pattern 推断
- 老师 spec 在哪——扫 course front_page、看 modules 树、点开最近 1-2 个 assignment 看链接，推断 spec 位置
- 作业命名 pattern 从历史 assignment 里 regex 出来

侦探跑完之后告诉学生："**我读了你这门课，我猜它是 X 类，spec 大概在 Y 那个 URL，作业一般叫成 'Project N' 这个格式。对吗？**" 学生 yes / no / 改改。

只有侦探**找不到 / 不确定**的字段，才主动问学生。每次问只问那一个具体字段，不集中堆问题。

这条彻底改变 onboarding 体感——不再是"我先答 10 个问题再开始"，而是"系统先告诉我它猜到啥，我确认或纠正一下就跑了"。

### 四个课型从空壳变成真有内容的 pipeline

**代码课**：clone 完之后系统真的会去 instructor 的网站抓 spec、下载 starter code、先写 test 再写实现、按 spec 里的数值约束自检（"不超过三个 import / 不超过 50 行"这种话被当成硬性 check），最后打包好让你 review 或自动交。今天这个 skill 只是个停下来报错的占位。

**写作课**：今天可以画注释 PDF 和填答题框，但写作部分是个一次成稿的 monolith。v0.1 把它拆成"研究 → 大纲 → 草稿 → humanize → verify"。humanize 阶段是个 first-class stage，但 v0.1 不会为它做大量 detector 调优——做完插件其他部分之后再回头深做。

**zyBook 数学课**：今天能渲染 LaTeX PDF，但 instructor 特殊要求（"每步引用法则要写出来"这类）是写死在 CEO 自己的 skill 里的。v0.1 把这种规则搬到学生自己的 design 文件里，系统按学生 instructor 的规则来。GradeScope 上传仍然手动。

**Canvas quiz**：今天 quiz 的 humanness 已经在跑，但写死在具体一门课里。v0.1 通用化——任何 Canvas quiz 都能用，配速窗口、目标分数段、是否故意错题这些都是学生在自己 design 文件里设的。

### 每个 skill 第一次跑能"边干边问"

如果 bootstrap 侦探没全部猜对，或者后来某个新作业系统识别不了，skill 不会 hard stop。它当场问那一道作业需要的几个具体参数，把答案累积写回 design 文件，继续做完这一道。**Design 文件随用随长**，不强求 day-1 完整。

### Verify 阶段会自动 loop 改进，不让人来回手动

每个 skill 的 verify stage 检查自己——长度数对了吗、约束都满足吗、写作课的话 detector 概率压下去了吗。如果某条没过，系统**自动跑回上一阶段重试**，重试有上限（默认 3 次）。3 次还过不去才停下来告诉用户"这一道我搞不定，要你介入"。

这条很重要——它把"verify 失败"从"故障"变成"产品自愈机制的一环"。学生不需要懂 detector 怎么工作；他只需要在 3 次 retry 都失败时才介入。

---

## v0.1 ship 之后产品仍然不干的事

- 不替你判断这门课用这个工具合不合学校政策。学术诚信 framework 自己定，工具不裁判
- 不主动 push 到任何远程，除非学生明确授权（见下面授权机制）
- 不发表"如何绕过 detector X"的研究内容
- 不支持 Canvas 之外的 LMS
- 不做 web UI、不做多租户托管

## 已定的产品决定（CEO 5.15 拍板）

### 降 AI 率深度调优——不在 v0.1 范畴

写作课的 humanize stage 跟其他 stage 一样存在、一样会跑、一样会 verify。但**不**为它单独做 detector 压测、不挑选 detector 名单去 ship、不写"我们对 X 有效"的承诺。等其他部分跑通了再回头深做。

v0.1 实际给的是机制：humanize 跑一遍 → verify 自己检查（local heuristic 或 detector API 都可以接） → 没过就自动 retry humanize，最多 3 次。机制对，效果以后调。

### Bootstrap 默认行为是"自己去找"，不是"先问学生"

四个 SKILL.md 的 Stage 0 和 bootstrap skill 都以这个为原则。学生唯一被打断的时机是：**侦探读完 Canvas 之后给出推测、请学生确认**，加上**侦探确实推不出某个字段时单点询问**。两种都是低摩擦交互。

任何 Canvas API 能拿到的字段都不该问学生（课名 / 课号 / module id / file id 这类）。Spec URL pattern、作业命名 regex、老师 rubric 这类需要内容理解的字段，侦探先尝试推断，把推断结果给学生 confirm。

### 代码课默认 Python 3.11

主流稳健、绝大多数学校 autograder 跑得动。worked demo 里写 3.11。具体课用 3.14 的，那门课的 design 文件单独覆盖即可。

### 写作课 voice register 用自由文本

学生写 "international student English with Chinese L1" 这种自由描述。不要 CEFR 标准。Humanize stage 把这段自由文本当 prompt 上下文。

### 自动提交：两层授权 + 问 trump 默认

design 文件里按**作业类型**写默认授权（"Project N: yes" / "Take-Home Exam: no" / "weekly Reading: yes"）。每次 canvas-scan 出 plan 时，对每个候选自动提交的作业**单独再问一次**。学生那一次的回答**永远 win**——比 design 更松（"这次也帮我交"）或更紧（"这次先不要"）都按学生当次说的算。

效果：design 是基线，scan 时是 ad-hoc 改写。学生从不会"被偷偷自动交"，因为每次都看到列表 + 单独问。

### License：GPL-3.0 + 明确邀请贡献

LICENSE 文件用 GPL-3.0——明确鼓励 fork 修改，但修改后必须也开源（防 fork 之后变闭源 SaaS）。

CONTRIBUTING.md 写明：欢迎别人贡献第五种 skill、欢迎贡献 worked demo overlay 给已有 skill、欢迎提 issue 报体验问题。

## 推进节奏

侦探式 bootstrap 是设计中心的转向，所以**先做它**，再回头写 4 个 SKILL.md（因为四个 SKILL.md 的 Stage 0 都依赖侦探机制）。

**第一周前半段**——重写 canvas-bootstrap：实现"读 Canvas → 推断 4 类 → 推断 spec 位置 → 推断作业命名 → 写 design 文件草稿 → 请学生 confirm"这个完整流程。这是 v0.1 onboarding 成败关键，先把它做对再做后面。

**第一周后半段 + 第二周前半段**——四个 per-course SKILL.md 改成真实可跑 pipeline。每个 skill 的 Stage 0 调用侦探机制读 design 文件、缺什么就当场点问。每个 stage 内部按 docs/SKILL_DESIGN.md 描述实现。每个 skill 在自己机器上跑真实作业验证。

**第二周中段**——找 1-2 个真人友盲测。另一个学校的学生 / 完全不懂 Canvas Pilot 的人，从 clone 开始跑一遍 setup + bootstrap + 一道真实作业。看哪里卡住、侦探猜得对不对、design 文件里推断的字段是否符合直觉。

**第二周后半段**——根据反馈调 bootstrap 推断逻辑、调 SKILL.md、调 README。

**第二周末**——orphan reset、push 到 X-isdoingreat、打 v0.1.0 tag、LICENSE + CONTRIBUTING 配齐、对外发出去。

humanize 深度调优 / detector 压测放到 v0.1 ship 之后下一个迭代。
